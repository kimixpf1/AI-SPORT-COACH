# 项目长期规则

- 始终保持整条分析链路稳定，优先保证上传、识别、分析、展示、部署全链路可用。
- 任何优化都要以不破坏现有核心功能为前提，先兼容训练视频上传与结果展示，再做增强。
- 每次完成修改后，必须先按真实用户路径完成模拟测试，重点覆盖微信手机端和电脑端。
- 真人模拟测试优先覆盖：选择动作、上传视频、等待分析、查看图表、查看建议、再次上传。
- 当前项目默认支持健身训练视频分析，重点覆盖深蹲、卧推、高翻等动作，并持续为更多动作预留扩展空间。
- 涉及 MediaPipe、轨迹分析、评分逻辑、结果卡片、历史记录、部署链路时，要联动检查上下游是否一起正常。
- 发现本地结果与线上结果不一致时，优先排查是否未推送、未部署、缓存未刷新、静态资源未更新或 Pages 构建失败。
- 新功能默认同时考虑手机端微信浏览器和桌面端浏览器体验，避免只对单端优化。
- 动作分析必须优先输出专业、可执行、用户能理解的反馈，避免空泛结论。
- 根目录固定维护 3 个文件：项目框架、todolist、项目迭代记录。
- 每次动手前，先更新 todolist 和项目迭代记录中的本次目标、当前状态、待办事项，必要时同步更新项目框架。
- 每次改动完成后，必须再次更新项目框架、todolist、项目迭代记录，写清本次改动、当前结构变化、遗留事项和下一步建议。
- 这些文件的目标是减少重复通读代码、避免遗忘已完成事项和用户已交代的长期要求。
- 一般使用 git 维护迭代记录，代码改完并验证无误后及时提交，提交前提仍是保证现有功能完整可用。
- 每轮改动完成并验证通过后，默认继续推进推送与部署准备，不额外等待提醒。
